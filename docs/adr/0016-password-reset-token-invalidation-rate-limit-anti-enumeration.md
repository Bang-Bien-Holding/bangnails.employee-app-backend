# Self-service password reset invalidates prior tokens on reissue, rate-limits by email and IP in Postgres, and answers every account state identically

Self-service password reset (#36) is the first public, unauthenticated endpoint in this codebase that accepts an arbitrary email address and does something observable with it. That combination — no auth, attacker-chosen input, a real side effect (an email sent or not) — is exactly the shape of an account-enumeration and email-bombing vector, so three things had to be decided together rather than bolted on after the fact: how the existing token machinery behaves when reused, how abuse is throttled without a new datastore, and how the HTTP response is shaped so none of that machinery is observable from outside.

## Decision

### Token invalidation lives inside the shared issuing function

`issuePasswordResetToken` (`internal/employees/service.go`) is the one function behind first-time activation (`CreateEmployee`), the admin-only bulk resend (`BulkSendPasswordResetLinks`), and the new self-service request (`RequestPasswordReset`). Before this work, it only ever inserted a new `password_reset_tokens` row — it never touched a still-valid row issued earlier for the same Employee, so multiple redeemable tokens could coexist indefinitely. An old, forwarded, or leaked link kept working after a fresh one was issued.

The fix — invalidate all of an Employee's prior unused, unexpired tokens, then insert the new one, in the same transaction — went inside `issuePasswordResetToken` itself rather than into each of its three callers. Fixing it per-caller would have meant three copies of the same invalidate-then-insert logic, with no guarantee a fourth future caller (or a modified existing one) remembers to include it; fixing it once, at the seam every caller already goes through, makes the guarantee unconditional. The two statements run in one transaction specifically so a mid-operation failure can't leave the Employee with the worst of both fixes attempted — zero redeemable tokens.

Making the fix universal exposed a second bug the single-token model always had but nothing previously triggered: two concurrent issuance calls for the same Employee (e.g. two admins clicking "resend" at once, or a self-service request racing an admin action) could both invalidate the prior token before either committed its insert, again leaving two valid tokens — the exact bug the transaction was meant to close, reintroduced by concurrency instead of by a missing step. A `KeyedMutex[int64]` (`internal/syncx`), locked on the Employee's id around the whole invalidate-then-insert transaction, serializes issuance per Employee without penalizing unrelated Employees issuing tokens at the same time.

No `token_type` column distinguishes an activation token from a reset token — see the confirmPassword section below for why that stays implicit.

### Rate limiting is a self-trimming Postgres table, dimensioned by email and by IP

`POST /v1/password-reset-requests` is throttled to 3 requests per submitted email per 15 minutes and 10 requests per source IP per 15 minutes, tracked in a dedicated `password_reset_requests` table (`ip_address INET`, `email TEXT`, `created_at`) written on every incoming request — including ones where the email matches no Employee at all.

**Why Postgres, not Redis**: at this system's current scale (~200 Employees), the same reasoning already applied to Sessions (see [ADR-0014](/docs/adr/0014-session-lifecycle-opaque-tokens-heartbeat-logout.md)) — a handful of indexed lookups and inserts per request is well within a single Postgres instance, and a second datastore isn't justified purely to hold a counter. It also keeps this feature's persistence in the same single-source-of-truth this codebase already uses everywhere else (Employees, Sessions, `password_reset_tokens`), rather than introducing a second kind of store with its own operational surface for one table.

**Why two dimensions, not just email**: the existing `password_reset_tokens` table only ever gains a row for an email that resolves to a real Employee, so it structurally cannot see or limit requests for emails that don't match any account — which is precisely the traffic an enumeration scan generates, and the case most worth limiting. `password_reset_requests` is written unconditionally, before the Employee lookup happens, so it can carry the IP dimension for exactly that traffic. Email and IP are independent checks; either one alone being over its threshold is sufficient to withhold the email — a single IP hammering many different addresses is caught by the IP limit even though no individual email is anywhere near its own.

**Why these thresholds and this window**: 3/email and 10/ip over 15 minutes are generous enough that a genuine Employee who mistypes their email or doesn't receive the first message can retry a couple of times without hitting the wall, while still being low enough to make a scripted scan slow and noisy. The 15-minute window is the single shared TTL used for both dimensions rather than two separate configurable windows, keeping the retention rule (below) simple: one width to trim against.

**Self-trimming, not a scheduled job**: on every call to `allowPasswordResetRequest`, rows older than the 15-minute window are deleted first, before the two counts are read and the new row is inserted. This mirrors nothing else in the codebase because nothing else needed it — Sessions and tokens are cleaned up by being redeemed, expiring in place, or superseded, not by a background sweep — but for a table that exists purely to answer "how many recently," a separate cron job would be an extra moving part for what a single `DELETE ... WHERE created_at < $1` already does inline, at the one place that reads the count anyway.

**Case normalization**: `employees.email` is `CITEXT` (case-insensitive) everywhere else in this codebase, but `password_reset_requests.email` is plain `TEXT`, so `alice@x.com` and `Alice@X.com` would otherwise count as two different rows — letting the per-email limit be bypassed with nothing more than re-casing the same address. `allowPasswordResetRequest` lowercases the submitted email before every count and insert, so the throttle tracks the same identity `employees.email` would resolve it to.

**Known gap, deliberately not closed here**: the delete, the two counts, and the insert are four separate round trips with no transaction or row lock across them, so a truly concurrent burst for the same email or IP can let slightly more requests through than the configured limit before every insert has committed. This is the same class of race the token-issuance `KeyedMutex` above closes for a different table — it wasn't applied here too, and is tracked separately (#42), because a rate limiter's job is to make abuse slow and costly, not to guarantee an exact count under adversarial concurrency; a few extra requests slipping through during a race changes the cost of an attack negligibly, whereas a stale reset token remaining redeemable is a correctness bug with no acceptable slack.

### The response is identical regardless of account state, except for malformed input

`POST /v1/password-reset-requests` always returns `200 OK` with the same generic message, whether the email is unknown, belongs to a deactivated Employee, belongs to an Employee who has never set a password, belongs to a fully active Employee, or was rejected by the rate limiter — none of these five cases produce a different status code or body. Every failure inside that branching (a DB lookup error, a mailer error, a rate-limit-check error) is logged and swallowed rather than surfaced, for the same reason: surfacing "transient failure" distinctly from "no such account" would itself be the enumeration signal this endpoint exists to avoid.

This deliberately covers status code and body only. The handler calls the service synchronously and writes the response only once that call returns, so the two branches that place a live call to the mailer (activation resend, password-reset send) take measurably longer than the three that return immediately (unknown email, inactive, rate-limited) — a response-timing side channel this ADR doesn't close. Issue #36's anti-enumeration requirement was scoped to the response itself (status/body), not to timing, so this is a known, accepted gap rather than an oversight; closing it would mean making the mailer call async from the handler's point of view, a larger change than this feature's scope justified for a guarantee nothing asked for.

Two exceptions to the generic response exist, and only one of them is deliberate. A syntactically malformed email is rejected with `400 Bad Request` before the service is ever invoked — pure input validation, depending only on the string's shape rather than on whether any account exists, so a distinct error here reveals nothing and costs nothing to allow. Separately, if the request's client IP can't be determined (`middleware.GetClientIPAddr` returns invalid), the handler returns `500` instead of the generic `200` — this is an infrastructure condition (a misconfigured reverse proxy, not attacker-controlled input), not a deliberate part of the anti-enumeration contract, but it is a second code path that doesn't return the uniform response.

Behind that uniform response, `RequestPasswordReset` still branches by Employee state, because the two real outcomes (send an email, or don't) depend on it:

- Unknown email → no email sent.
- Found, `is_active = false` → no email sent.
- Found, active, no password ever set (pending activation) → the activation email is resent, not a reset email — an Employee who forgot they never finished activating gets routed to the flow that actually applies to them, without the endpoint needing to say so.
- Found, active, password set → a password-reset email is sent.

### `confirmPassword` is a field, not a new endpoint or a token-type column

`POST /v1/activate` already redeemed a `password_reset_tokens` row and set a bcrypt hash — the same operation whether the token came from first-time activation, an admin-triggered resend, or this new self-service flow. Self-service completion reuses that endpoint unchanged in route and purpose, gaining one new required field, `confirmPassword`, checked server-side via `eqfield=Password` — a mismatch fails validation the same way any other malformed field does, so no new response shape or error path was needed. Enforcing this server-side rather than trusting the frontend alone matches this system's existing posture of not treating client-side validation as sufficient on its own (the same reasoning behind server-side email-format and password-length checks on this same endpoint).

No `token_type` column was added to distinguish "this token is for activation" from "this token is for a reset." The distinction stays exactly as implicit as it already was: `RequestPasswordReset` infers which email to send from whether `employee.Password` is already set at request time (mirrors `auth.Service.Login`'s own `len(Password) == 0` check for "not yet activated"), and completion doesn't need to know which kind of token it's redeeming at all — setting a bcrypt hash, resetting lockout state, and deleting the Employee's Session are the identical operation regardless of which email sent the link. That Session deletion is a fourth Session-ending trigger alongside the three [ADR-0014](/docs/adr/0014-session-lifecycle-opaque-tokens-heartbeat-logout.md) already documents (expiry, Logout, Heartbeat failure) plus supersession by a new Login — this ADR is the record of that addition to the Session lifecycle, not a change ADR-0014 itself needed to make, since a completed reset isn't a Login or Heartbeat outcome.

This is safe only because, today, a password can be set through exactly one code path in this system: this token-redemption flow. If a future feature introduces a second way to change `employees.password` outside of it — the spec names an admin "force password reset" that clears the field directly as the concrete example — the implicit signal breaks: an Employee an admin just force-reset would look, from `RequestPasswordReset`'s point of view, identical to one who has never activated, and would get an activation-flavored email instead of a reset-flavored one. That is the condition under which a `token_type` column (or equivalent explicit signal) needs to be introduced; it isn't needed for anything this feature does on its own.

## Considered Options

- **Per-caller token invalidation** (each of `CreateEmployee`, `BulkSendPasswordResetLinks`, `RequestPasswordReset` invalidates prior tokens itself before calling the issuing function) — rejected: three copies of the same logic, with no structural guarantee a new caller includes it, versus one copy at the seam every caller already passes through.
- **Redis-backed rate limiting** — rejected for the same reason Sessions rejected it in [ADR-0014](/docs/adr/0014-session-lifecycle-opaque-tokens-heartbeat-logout.md): no second datastore is justified at ~200 Employees for a counter a single Postgres table answers comfortably.
- **A single rate-limit dimension (email only)** — rejected: the existing token table already can't see unmatched emails, and email-only limiting would leave IP-based scanning of unregistered addresses completely unthrottled, which is the traffic this feature most needs to slow down.
- **A `token_type` column on `password_reset_tokens` now** — rejected as premature: nothing in this system can currently set or clear a password outside the one token-redemption path, so the implicit signal is sound today; adding the column ahead of a concrete second path would be speculative schema for a condition that doesn't yet exist.
- **A distinct completion endpoint for self-service reset** — rejected: the operation (redeem token, set hash, clear session/lockout) is identical to what `POST /v1/activate` already did; a second endpoint would duplicate that logic for no behavioral difference.

## Consequences

- Every call to `issuePasswordResetToken` now takes a per-Employee lock and runs one extra `UPDATE` inside its transaction — negligible at this system's request volume, but it means issuance is no longer a single-statement operation for any of its three callers.
- `password_reset_requests` grows and shrinks with live traffic rather than being bounded by a scheduled job; a sustained attack sized to stay just under the cleanup's lookback window would keep the table larger than idle usage would, though still bounded by the window width, not unbounded.
- The rate limiter's known concurrency gap (#42) means the 3/15min and 10/15min figures are a strong deterrent, not a hard ceiling — anyone relying on this table for an exact audit count of requests would be relying on a guarantee it doesn't make.
- If a future admin-facing "force password reset" (or any other path to clearing/setting `employees.password` outside token redemption) ships, `RequestPasswordReset`'s activation-vs-reset branch and this ADR's confirmPassword section both need revisiting before that feature ships, not after.
- Every error path inside `RequestPasswordReset` and `allowPasswordResetRequest` is now logged-and-swallowed by design — a future maintainer adding a new branch here must resist the instinct to surface a clearer error to the caller, since a clearer error is exactly the enumeration leak this endpoint exists to prevent.
- The anti-enumeration guarantee covers response status/body only, not latency: the two branches that place a live mailer call are observably slower than the three that return immediately, and an undeterminable client IP breaks the uniform-response contract outright with a `500`. Neither gap is closed by this feature; a future hardening pass should treat both as open, not assume the generic-`200` design already covers them.
