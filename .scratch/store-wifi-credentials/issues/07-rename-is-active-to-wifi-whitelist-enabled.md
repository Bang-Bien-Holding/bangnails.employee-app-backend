# 07 — Rename store.is_active to wifi_whitelist_enabled; remove it from PATCH /v1/stores/{id}; stop sync from force-setting it

**What to build:** Rename `store.is_active` to `store.wifi_whitelist_enabled` in place, narrow its documented effect to gating only the wifi-whitelist login check (not the GPS/geofence fallback), stop `UpsertStores` from unconditionally resetting it to `true` on every Odoo sync, **and remove the field from `PATCH /v1/stores/{id}`'s writable set** (it moves to its own endpoint — ticket 10 — and to a rewritten bulk endpoint — ticket 09). See [ADR-0004](/docs/adr/0004-store-wifi-whitelist-enabled-replaces-is-active.md) and [ADR-0006](/docs/adr/0006-single-store-wifi-toggle-endpoint-bulk-becomes-atomic.md).

**Blocked by:** 05 (built the `is_active` toggle on `PATCH /v1/stores/{id}` that this ticket both renames the underlying column for and removes from that endpoint)

**Status:** done

- [ ] New migration (`000NN_rename_store_wifi_whitelist_enabled.sql`): `ALTER TABLE store RENAME COLUMN is_active TO wifi_whitelist_enabled;` (up), reverse rename (down). Column keeps its existing `NOT NULL DEFAULT true`.
- [ ] `UpsertStores`' `ON CONFLICT (odoo_store_id) DO UPDATE` clause drops `is_active = true` entirely — on conflict, only `store_name`/`city` are updated. The sync no longer writes to this column in the reactivation direction (see ticket 08 for the removal direction).
- [ ] All Go references renamed: `sqlc` models/queries regenerated (`IsActive` → `WifiWhitelistEnabled` field, `is_active` → `wifi_whitelist_enabled` in generated SQL), `internal/stores/types.go` (`StoreDetail.IsActive`), `internal/stores/service.go`, `internal/stores/handlers.go`.
- [ ] **`patchStoreParams.IsActive` is removed, not renamed.** `UpdateStore`'s params and the underlying geofence/wifi-list update logic no longer accept or touch `wifi_whitelist_enabled` at all — ticket 05's PATCH-scope addition is undone here rather than carried forward under the new name. `PATCH /v1/stores/{id}`'s request body loses the field entirely; its **response** body still reports `wifi_whitelist_enabled` (unchanged shape, now read-only from this endpoint's perspective — see tickets 09/10 for how it's set).
- [ ] `GET /v1/stores/{id}` and `GET /v1/stores` response body field renamed `is_active` → `wifi_whitelist_enabled` (read-only reporting, unaffected by the PATCH-scope removal above).
- [ ] Doc comments referencing `is_active`/soft-delete (`ErrStoreNotFound`, the `UpsertStores`/`UpdateStoreGeofence` query comments citing ADR-0001) updated to reference the renamed column and ADR-0004/ADR-0006.
- [ ] `CONTEXT.md`'s Store glossary entry updated: `wifi_whitelist_enabled` replaces `is_active`, effect narrowed to "gates the wifi-whitelist login check only; the GPS/geofence fallback is unaffected."
- [ ] `.scratch/store-wifi-credentials/spec.md` updated: field renamed throughout (Solution, GET body examples, Implementation Decisions), narrowed-scope wording added, PATCH /v1/stores/{id} scope narrowed (already reflected in the current spec revision — verify it matches what ships here).
- [ ] Existing tests asserting `is_active` (service and handler seams, `TestStoreService_SyncStores`'s reactivation assertions) updated to assert `wifi_whitelist_enabled` and the new "sync doesn't touch it" behavior — a store with `wifi_whitelist_enabled = false` before a sync run still has it `false` after, even though it's present in Odoo's response.
- [ ] Existing `UpdateStore` tests covering `is_active` present-vs-omitted (from ticket 05) are **removed**, not renamed — that behavior no longer exists on this endpoint. `SyncStores` tests updated to confirm it no longer flips an existing store's flag to `true`.
- [ ] Handler seam: existing PATCH/GET response-shape tests updated — GET keeps asserting the (renamed) field is present; PATCH request-body tests drop any `wifi_whitelist_enabled`/`is_active` input coverage, response-shape assertions keep checking the field is present read-only.
