# 09 — PATCH /v1/stores (bulk wifi_whitelist_enabled toggle, atomic + locked)

**What to build:** A collection-level bulk endpoint that atomically sets `wifi_whitelist_enabled` on a set of stores at once — the admin-facing "deactivate wifi for all stores" (or any explicit subset) action. Rewritten from its original best-effort, lock-exempt design — see [ADR-0006](/docs/adr/0006-single-store-wifi-toggle-endpoint-bulk-becomes-atomic.md), which supersedes the endpoint-shape portion of [ADR-0004](/docs/adr/0004-store-wifi-whitelist-enabled-replaces-is-active.md).

**Blocked by:** 07 (renames the field this endpoint sets; also establishes the single-store PATCH shape ticket 10 mirrors)

**Status:** done

- [ ] `PATCH /v1/stores` (collection URL, no `{id}`) registered in the router, distinct from `PATCH /v1/stores/{id}` and `PATCH /v1/stores/{id}/wifi-whitelist-enabled`.
- [ ] Request body: `{"stores": [{"id": 1, "updated_at": "..."}, {"id": 2, "updated_at": "..."}], "wifi_whitelist_enabled": false}`. `stores`: required, non-empty, `dive`-validated (`validate:"required,min=1,dive"`), each element's `id` and `updated_at` also `required`. `wifi_whitelist_enabled`: required non-pointer bool.
- [ ] **Optimistic-locked and atomic, per store** — this is the reversal from the original design. Inside one transaction: (1) fetch current `(id, updated_at)` for every submitted id (`SELECT id, updated_at FROM store WHERE id = ANY($1::bigint[])`); (2) compare against the submitted `(id, updated_at)` pairs in the service — any missing id or any `updated_at` mismatch means the *entire* request is rejected before any write, with every mismatched/missing id collected into `failed_ids`; (3) only if every id matched, a bulk `UPDATE store SET wifi_whitelist_enabled = $1, updated_at = now() WHERE id = ANY($2::bigint[]) RETURNING id, updated_at` to apply the change and return fresh state.
- [ ] No partial application — an unknown id or one stale `updated_at` in the batch fails the whole request (`409`), unlike `DELETE .../wifi-whitelist`'s best-effort model. Not-found and stale-lock aren't distinguished in the failure body since the client's remedy (re-fetch `GET /v1/stores`, retry) is the same either way.
- [ ] Response on success (`200`): a JSON array, one element per store, with fresh state: `[{"id": 1, "wifi_whitelist_enabled": false, "updated_at": "..."}]`.
- [ ] Response on failure (`409`): `{"failed_ids": [2]}` — no partial results array.
- [ ] `Service` gains a new/changed method, e.g. `BulkSetWifiWhitelistEnabled(ctx, params BulkSetWifiWhitelistEnabledParams) ([]StoreDetail, error)` — signature changed from the original best-effort `[]BulkActionResult` design; returns the sentinel conflict error (reusing `ErrStoreConflict`) carrying the `failed_ids` list on any mismatch.
- [ ] The old `BulkActionResult`-shaped per-id result type from the original design is **not** used here — this endpoint's response types are local to this ticket (success: `[]StoreDetail`-derived array; failure: `{failed_ids: []int64}`).
- [ ] Service seam: tests cover all-ids-match-and-exist succeeding with fresh state returned for every store; one stale `updated_at` among several valid ids aborting the whole call with zero side effects, reporting every mismatched/missing id in `failed_ids`; one unknown id among several existing ones also aborting the whole call (not a partial success).
- [ ] Handler seam: tests cover `400` for empty/missing `stores` or missing `wifi_whitelist_enabled`, `409` with the `failed_ids` body shape for any unknown id or stale `updated_at` in the batch, and the success response's per-store array shape.
